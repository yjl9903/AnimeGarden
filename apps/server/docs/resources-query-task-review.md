# Resources Query Task 设计预期与现状核对

日期：2026-03-22

## 背景

`resources` 查询模块面对的是复杂筛选 + 分页的场景。完整筛选条件很多，直接对完整查询做缓存时，命中率往往不高；但真实业务中，用户通常会先通过一个主关键词、标题匹配、subject、类型或字幕组等条件筛出一批有限结果，然后再继续细化筛选。

基于这个特点，模块引入了 `Task` 类和预取机制，希望把查询拆成两层：

1. 先用一组更粗的关键条件查询数据库，得到可以复用的中间结果。
2. 把这批中间结果缓存到进程内存中。
3. 对同一个粗查询下的后续细化请求，优先走内存筛选，而不是再次直接打数据库。
4. 只有内存缓存不足以判断最终分页结果时，才继续向后预取更多数据。
5. 如果预取达到上限，但底层数据库里仍可能存在更多结果，则回退到直接查数据库，保证结果正确性。

目标不是完全替代数据库查询，而是尽量提高“同一类查询上下文中的复用率”，从而降低复杂筛选分页场景下的数据库压力和响应时间。

## 设计预期

当前模块的设计预期可以总结为：

### 1. Task 需要抽取“粗查询”

`Task` 的 key 不应该等价于完整过滤条件，而应该只保留最有代表性的主筛选条件，例如：

- `search`
- `include`
- `subjects`
- `types`
- `fansubs` / `publishers`

这样后续更细的过滤可以复用同一个 `Task` 的中间结果。

### 2. Task 结果需要缓存到内存

一次 `Task` 预取得到的结果应当缓存在内存里，供：

- 同一个请求的后续分页使用
- 同一时间的并发请求复用
- 后续相同粗查询、但更细过滤不同的请求复用

### 3. 内存筛选需要承担“细化过滤”

完整过滤条件中的一部分需要在数据库阶段完成，另一部分则应在 `Task` 缓存结果上做内存筛选，以降低数据库重复执行复杂查询的概率。

### 4. 预取必须是按需的

预取不能因为来了一个请求就无条件反复往后拉更多页。只有在“当前缓存不足以筛选出目标页最终结果”时，才允许继续向后预取。

### 5. 预取必须有总上限

为了避免单个 `Task` 无限扩张，占用过多内存或不断把大批数据从数据库搬到内存中，预取需要有一个较大的总上限。达到上限后，如果数据库仍可能有更多数据，就回退到直接数据库查询。

## 当前实现已达到的部分

基于 2026-03-22 的代码，当前实现已经满足以下几点：

### 1. 已存在粗查询到 Task 的分层

`findFromTask()` 会优先把完整查询压缩成更粗的 `Task` 选项，再用完整查询选项在内存中做细筛。

当前粗查询分组包括：

- `search`
- `include`
- `subjects`
- `types`
- `fansubs` / `publishers`
- 其余情况回退为完整过滤条件

### 2. Task 结果会缓存在进程内存中

`Task.resources` 会保存预取结果。后续请求若命中同一个 `Task`，可以复用这批中间结果，而不是重新从数据库取同一批粗结果。

### 3. 并发请求会复用同一轮预取

当前实现把“按游标预取”的进行中 Promise 缓存在 `Map<number, Promise<void>>` 里。多个请求同时命中同一 `Task`、同一游标时，会等待同一轮预取，而不是并发重复打数据库。

### 4. 预取是按需推进的

当前 `findFromTask()` 的流程已经变成：

1. 先尝试用当前缓存做内存筛选。
2. 如果当前缓存已经足够筛出目标页，则直接返回。
3. 只有当前缓存不足，并且 `Task` 仍允许继续预取时，才向后按游标继续预取。

这符合“只有在收到的数据不够筛选出最终结果时，才预取下一页”的预期。

### 5. 预取存在总上限，超限后会 fallback 到 DB

当前已经引入了总预取上限：

- `RESOURCES_TASK_PREFETCH_MAX_COUNT = 5000`

当某个 `Task` 的内存预取达到上限，但数据库中仍可能存在更多数据时，`Task` 会停止继续扩容，此时 `findFromTask()` 会退出 task 路径并回退到直接查数据库。

### 6. 当前行为已有测试覆盖

目前测试已覆盖至少两类关键行为：

- 并发请求与后续请求对 `Task` 预取结果的复用
- 达到总预取上限后的停止扩容行为

## 当前实现尚未完全达到预期的部分

虽然整体机制已经基本成型，但若以“优化复杂筛选分页查询，并最大化细化查询的复用率”为目标，当前实现仍有几个关键偏差。

### 1. Task key 仍然偏细，缓存复用率还不够高

当前实现中：

- `search` task 仍然带上了 `exclude`
- `include` task 仍然带上了 `keywords` 和 `exclude`

这意味着一些本应共享同一个粗查询结果的请求，会被拆成多个不同 `Task`，从而降低中间结果复用率。

如果设计目标是“先按主关键词/主条件打粗缓存，再在内存里做细化过滤”，那么这部分还可以继续收敛。

### 2. 内存筛选语义与数据库筛选语义不完全一致

数据库层在 `include / keywords / exclude` 上使用的是 `titleAlt` 的 `ILIKE` 匹配；但当前内存筛选使用的是 `normalizeTitle(r.title).toLowerCase()`。

这带来一个问题：

- task 路径下的内存筛选结果
- 直接 fallback 到数据库时的 SQL 结果

二者理论上可能不完全一致。

这会削弱整个优化方案的可靠性，因为 `Task` 机制的前提之一就是“内存细筛应尽量等价于数据库细筛”。

### 3. Task 缓存和 fallback 之间还存在永久降级问题

当前 `downgrade` 是一个长期存在的集合。某个完整查询一旦走过 fallback，就会被加入 `downgrade`，后续相同完整查询会持续跳过 task 路径，直接查数据库。

这会让某些查询在经历一次 fallback 后，失去后续继续享受 `Task` 缓存优化的机会。

如果预期是“同类查询在后续访问中仍尽可能复用粗结果缓存”，那么这个降级机制目前还过于激进。

## 结论

以 2026-03-22 的实现来看，这个模块已经基本具备了原始设计思路的主体结构：

- 有 `Task` 这一层来承载中间结果
- 有内存缓存
- 有并发复用
- 有按需向后预取
- 有总预取上限
- 有超限 fallback 到 DB 的保护逻辑

因此，从“模块机制是否已经落地”这个角度看，答案是：**基本达到预期**。

但如果从“是否已经充分实现最初设计目标，即显著优化复杂筛选分页场景，并稳定提升细化筛选的复用率”来看，答案是：**部分达到，仍未完全达到**。

当前最主要的剩余问题不是预取框架本身，而是：

1. `Task` 粗查询的粒度仍可继续收敛。
2. 内存筛选与数据库筛选的语义还需要对齐。
3. fallback 后的降级策略还需要变得更可恢复，而不是长期绕过 task 机制。

## 后续建议

建议下一步优先处理以下三件事：

### 1. 继续收敛 Task key

目标：让同一主查询上下文下的更多细化请求可以命中同一个 `Task`。

### 2. 对齐内存筛选与 DB 筛选语义

目标：保证 task 路径和 fallback 路径在结果上尽量一致，避免“缓存优化改变结果语义”。

### 3. 把 downgrade 从永久绕过改成可恢复的负缓存

目标：避免一次 fallback 之后，同类查询长期失去 `Task` 复用收益。
